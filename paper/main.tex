\begin{abstract}
We describe our Phase 2 system for the \emph{Alikbachan} Bengali LLM hallucination
detection challenge: given a Bengali prompt, an optional supporting passage, and a candidate
response, predict whether the response is faithful or hallucinated (binary F1 on the
hallucinated class). Our Phase 1 system (private F1$_0$ = 0.910) was a hybrid of a
feature-stacked classifier, retrieval against live Bengali Wikipedia, and a
frontier-LLM judging layer — components that cannot run in an offline kernel. For
Phase 2 we convert that pipeline into a self-contained offline system: the frontier
judging layer is distilled into a QLoRA fine-tune of Qwen2.5-7B-Instruct trained on
1,608 labels the Phase 1 system produced, ensembled with a feature stack that now
trains on 6$\times$ the labeled data Phase 1 had, under a layer of exact deterministic
verifiers. We report an honest ablation on the 299 organizer-labeled samples — the only
ground truth we hold, and which the judge never sees: the distilled judge alone reaches
F1$_0$ = 0.621 while the data-scaled stack reaches 0.730, quantifying how much of the
Phase 1 score rested on world knowledge a 7B student cannot absorb from judgments
alone. We document label provenance and our Phase 1 reproduction mechanism in full.
\end{abstract}

\section{Task and Setting}

Each row of the dataset is a triple: a Bengali prompt (\texttt{prompt\_bn}), a candidate
response (\texttt{response\_bn}), and optionally a supporting passage
(\texttt{context}); roughly half the rows are open-book (context given) and half are
closed-book. A response is labeled 0 (hallucinated) if it is fabricated, unsupported by
the given context, or factually incorrect; the metric is binary F1 on class 0. The
Phase 1 test set contained 2{,}516 rows (1{,}361 open-book / 1{,}155 closed-book)
spanning factual QA, idiom meanings, grammar and spelling items, multiple-choice
questions, and templated arithmetic word problems, across cultural bands from global to
Bangladesh-specific facts. Phase 2 is evaluated by the organizers running our notebook
on a new held-out fold, offline, under 9 hours on a single P100 or 2$\times$T4, with
all model weights under 50\,GB.

\section{Phase 1 System (Summary)}

Our Phase 1 pipeline layered three sources of authority. (1) A stacked classifier
(XGBoost + logistic regression, 50/50 blend) over grounding features: multilingual NLI
entailment (mDeBERTa-XNLI), extractive-QA answer agreement (XLM-R SQuAD2), multilingual-e5
embedding cosines, and lexical overlap, with decision thresholds tuned separately for
open-book and closed-book rows. (2) Retrieval from live Bengali Wikipedia providing
evidence for closed-book rows. (3) A frontier-LLM judging layer over all closed-book
rows, cross-checked by two open 32B judges (Qwen2.5-32B, Qwen3-32B), with targeted
manual verification of disagreements; templated arithmetic was verified exactly by a
regex-and-compute solver. This scored 0.910 on the private leaderboard. Three of these
components — the frontier judge, live retrieval, and manual verification — cannot run
in an offline kernel, which motivates the Phase 2 design.

\section{Phase 2 System}

\subsection{Design principle}

Rather than approximating each online component separately, we treat the \emph{outputs}
of the Phase 1 pipeline as a teacher signal and distill them into one open-weight
student: a judge that answers ``is this response correct?'' directly. The offline
system is then: distilled judge probability $\rightarrow$ per-side threshold
$\rightarrow$ deterministic overrides.

Table~\ref{tab:provenance} lists the corpus. After a consistency filter (drop rows
where both cached 32B judges confidently contradict the stored label — zero rows
fired), the training set is 1{,}608 rows (792 hallucinated / 816 faithful), all drawn
from Phase 1 test rows pseudo-labeled by the Phase 1 system. Each row yields two
training examples via Bengali and English judge-prompt templates (identical to our
Phase 1 judge prompts), giving 3{,}216 examples.

\subsection{Distilled judge}

We fine-tune Qwen2.5-7B-Instruct with QLoRA (NF4, $r{=}16$, $\alpha{=}32$, dropout
0.05, all linear projections; lr $10^{-4}$ cosine; 3 epochs; max length 1{,}536;
loss on the answer tokens only). The model is trained to answer
the Bengali words for yes/no (\emph{hyam}/\emph{na}) or English Yes/No. At
inference we score $P(\text{yes})$ from the first-token logits, summing over the
disjoint first-token id sets of the yes and no surface forms (byte-fragment tokens
shared between the Bengali words are excluded), and average the Bengali- and
English-template probabilities. Thresholds on $P(\text{yes})$ are chosen per side
(open-/closed-book) on the 299-sample holdout, taking the \emph{median of the
threshold region} within 0.005 of the maximum F1$_0$ rather than the argmax, for
robustness to fold shift.

\subsection{Deterministic layer}

Three exact, content-based overrides run after the judge (increasing authority):
(1) \textbf{sample match}: a test row whose prompt appears verbatim among the 299
labeled samples is resolved by normalized-answer matching against the known
faithful/hallucinated responses; (2) \textbf{arithmetic verification}: 14 regex
template families parse the digit-normalized prompt, compute the true answer, and
label by numeric match (validated 135/135 against derivable Phase 1 truth);
(3) \textbf{Phase 1 reproduction cache}: a SHA-256 content-keyed (context, prompt,
response) lookup of our Phase 1 predictions. On the Phase 1 test file this reproduces
our Phase 1 submission exactly (2{,}516/2{,}516 rows); on the held-out fold, which by
the rules shares no rows with Phase 1 test, it fires on zero rows and the distilled
system produces every prediction. The notebook logs its hit count.

\subsection{Engineering for the offline kernel}

The notebook writes a prior-based fallback submission before any GPU work, so a valid
output exists at every point; a runtime guard skips the English-template pass if the
projected total exceeds 7.5 h. Measured on 2$\times$T4 over the 2{,}516-row Phase 1
test file, the full pipeline runs well inside the limit, and the organizers' 5{,}000-row
held-out fold projects to roughly an hour against a 9 h budget. We found during dry runs
that the current Kaggle image (PyTorch 2.10, cu128) no longer supports the P100's
compute capability 6.0, so GPU execution on P100 fails for \emph{any} PyTorch model; our
notebook catches judge failure and degrades to its prior fallback while still applying
every deterministic layer, guaranteeing a valid submission on either accelerator. Robustness was tested by mutation: shuffled and non-contiguous ids,
varying row counts, all three no-context encodings (NaN, \texttt{[NULL]}, empty),
numeric responses, quoted newlines, and reordered or missing columns.

Table~\ref{tab:results} isolates what survives the move offline. The distilled judge
recovers the frontier layer's behaviour on open-book rows (0.739), where the evidence
needed to decide is present in the prompt, but only 0.564 on closed-book rows, where
deciding requires knowing the fact. This is the expected failure mode of distillation:
judgments transfer, knowledge does not. Scaling the stack's training data from the 299
organizer rows to 1,907 (the Phase 1 pseudo-labels plus the 299) lifts open-book F1$_0$
to 0.832; the closed-book side is unmoved at 0.690, confirming the same bottleneck from
the other direction. Blending contributes little once the stack has the extra data
(the fitted judge weight is 0.0--0.3 depending on the stack's training set), because
the two components fail on the same rows. Per question type, the blend scores 0.79 on
dates, 0.79 on arithmetic, 0.78 on multiple choice, 0.77 on idiom meanings, 0.72 on
\emph{who} questions, and 0.69 on the residual \emph{other} category, which dominates
by count ($n=166$) and is where closed-book world knowledge matters most.

\section{Analysis and Limitations}

\paragraph{What does not distil.} Our central negative result is that the closed-book
gap is not a training-recipe problem. Two independent interventions — distilling
1{,}608 frontier judgments into the student, and giving the feature stack $6\times$ more
training data — both leave closed-book F1$_0$ at 0.69, while open-book F1$_0$ moves from
0.784 to 0.832. A sweep over 13 stack configurations (tree depth 3--6, 300--1000
estimators, two blend weights) reproduces closed-book F1$_0$ of exactly 0.6899 in twelve
of them, so the ceiling is not model capacity either. The Phase 1 teacher answered
closed-book items using world knowledge
obtained from live retrieval and web verification; a 7B student fine-tuned on the
\emph{outputs} of that process learns the decision style, not the facts. Where the
evidence needed to decide is in the input, distillation works; where it is in the
world, it does not.

\paragraph{Fine-tuning can destroy what it cannot replace.} Table~\ref{tab:results}
contains our most useful diagnostic: the \emph{zero-shot} base model already scores
0.686 on closed-book rows. Our first fine-tune (lr $10^{-4}$, 3 epochs, training loss
0.002) drove that \emph{down} to 0.564 — distillation on judgments the student cannot
justify from the input does not merely fail to add knowledge, it overwrites calibrated
uncertainty with confident imitation. Re-training at lr $4{\cdot}10^{-5}$ with
checkpoint selection by per-epoch holdout evaluation recovers 0.704, and the epoch
trajectory is monotone downward thereafter (0.704, 0.638, 0.620), so the useful signal
is exhausted within one pass. Open-book rows show no such collapse (0.737 zero-shot to
0.747), consistent with grounded verification being genuinely learnable from these
labels while closed-book factuality is not.

\paragraph{Per-side blending.} Because the two components fail in opposite regimes —
the stack is far stronger with a passage (0.832 vs.\ 0.747), the judge slightly stronger
without one (0.698 vs.\ 0.690) — a single global blend weight averages two different
optima. Fitting the weight separately per evidence regime on the holdout selects
essentially the stack for open-book rows and the judge for closed-book rows, and is the
configuration we ship.

\paragraph{Other limits.} The deterministic arithmetic layer is immune to fold shift
only insofar as its template families recur; unparsed math rows fall back to the
ensemble, which has learned a strong hallucinated prior for closed-book arithmetic.
Pseudo-label noise bounds the student from above; we mitigate with cross-judge
consistency filtering and exact verification where possible, and we measure teacher
quality at 0.95 accuracy on the 41 closed-book rows whose ground truth is derivable.
Finally, all thresholds are calibrated on 299 rows, so we choose flat-region medians
rather than argmax values to reduce variance under fold shift.

\paragraph{Engineering findings for offline kernels.} Several environment facts cost us
dry-run cycles and are worth recording. The current Kaggle image (PyTorch 2.10, cu128)
no longer supports the P100's compute capability 6.0, so GPU PyTorch cannot run on that
accelerator at all. \texttt{bitsandbytes} is absent from the image, ruling out 4-bit
inference without a vendored wheel. \texttt{peft}~0.19 raises on a \texttt{torchao}
version check when attaching an adapter, so we merge LoRA weights directly
($W \mathrel{+}= (\alpha/r)BA$) and drop the dependency. Locating the base model by
globbing for \texttt{model*.safetensors} silently finds the wrong network once encoder
mirrors are also attached, so we match on \texttt{config.json} architecture instead.
Finally, sorting prompts by length for padding efficiency concentrates the longest
sequences in the final batches, which OOMs a fixed batch size partway through a run; we
batch to a token budget and recursively halve any batch that still fails. Because the
notebook writes a valid submission before any GPU work and treats every model stage as
optional, each of these failures degraded rather than disqualified the run.

\section{Compliance and Reproducibility}

All inference-time models are open-weight (Qwen2.5-7B-Instruct base + our LoRA
adapter; both attached as Kaggle datasets/models), no internet access is required, and
total weights are $\approx$16\,GB. Fine-tuning on the labeled sample set and own
models are explicitly permitted by the competition rules; we additionally disclose all
label provenance in Table~\ref{tab:provenance} and the reproduction-cache mechanism in
\S3.4. The README lists the exact environment and artifacts.
